- **feat(summarize)**: Phase 11 ‚Äî `summarizeUnsummarized()` query rewritten with a value-first pickup policy. Adds `ORDER BY (relevance + importance) DESC, published_at DESC`, a 14-day recency window (`SUMMARIZE_MAX_AGE_DAYS`), and an `eq(feeds.enabled, true)` filter. **Why**: the previous query had no `ORDER BY` at all ‚Äî Postgres returned rows in arbitrary order (~insertion order), so under queue pressure the oldest-analyzed articles got summarized first while high-relevance items sat in the queue. Unlike analyze (which sorts by timestamp because it has to guess by age), summarize already has the LLM scores and can pick by measured value. Degradation shape under overload: lowest-value articles accumulate in state 2 and age out at 14 days without being summarized ‚Äî which is correct, because they'd never appear in the dashboard anyway (low composite score). No per-feed allocation needed; analyze's distribution already balanced state 2. Design rationale in [pipeline-flow.md](pipeline-flow.md) Layer 3.
- **fix(analyze)**: Phase 10.1 ‚Äî replaced the post-allocation freshness-desc sort with **round-robin interleaving across feed buckets**. **Why**: the freshness-desc sort I shipped in Phase 10 caused arXiv's hyper-fresh `published_at` timestamps to cluster at the top of the iteration, so when the user cancelled a run after 30 articles, all 30 were arXiv (14 cs.AI + 14 cs.CL + 2 cs.LG) and lab feeds got 0 ‚Äî same starvation symptom as the original bug, just one step later in the pipeline. Round-robin now takes one article from each feed bucket per round, ordered by weight desc, so cancelling mid-run leaves every feed with at least `floor(processed / feedCount)` articles. Within each feed, articles stay newest-first via the per-feed query's `ORDER BY`. Memo updated (Q4 answer flipped). My original Q4 vote was wrong ‚Äî caught on the first real test run via DB inspection of the cancelled run's distribution.
- **feat(analyze)**: Phase 10 ‚Äî analyze allocation now distributes the batch across feeds by weight with spillover, replacing the broken `ORDER BY pub DESC LIMIT 100` query. **Why**: the first post-Phase-9 diagnostic run analyzed 99/100 articles from arXiv cs.AI alone while every lab feed (Anthropic, DeepMind, Meta AI, NVIDIA, Microsoft Research, Mistral, OpenAI) got zero coverage. Per-feed allocation logged via `[analyze] allocation: feed=N/pending ‚Ä¶` for run-time visibility. **Behavior change**: disabled feeds (`enabled = false`) now also skip analyze, not just fetch ‚Äî matches the `analyze_weight = 0` semantics. Phase 10 design memo: [phase10-analyze-allocation-memo.md](phase10-analyze-allocation-memo.md). Validated by a realistic 12-feed allocation scenario test that mirrors today's actual production data ‚Äî all 12 feeds get representation, no single arXiv feed exceeds 20 slots.
