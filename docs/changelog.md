## 2026-04-14

- **feat(summarize)**: Phase 11 — `summarizeUnsummarized()` query rewritten with a value-first pickup policy. Adds `ORDER BY (relevance + importance) DESC, published_at DESC`, a 14-day recency window (`SUMMARIZE_MAX_AGE_DAYS`), and an `eq(feeds.enabled, true)` filter. **Why**: the previous query had no `ORDER BY` at all — Postgres returned rows in arbitrary order (~insertion order), so under queue pressure the oldest-analyzed articles got summarized first while high-relevance items sat in the queue. Unlike analyze (which sorts by timestamp because it has to guess by age), summarize already has the LLM scores and can pick by measured value. Degradation shape under overload: lowest-value articles accumulate in state 2 and age out at 14 days without being summarized — which is correct, because they'd never appear in the dashboard anyway (low composite score). No per-feed allocation needed; analyze's distribution already balanced state 2. Design rationale in [pipeline-flow.md](pipeline-flow.md) Layer 3.
- **fix(analyze)**: Phase 10.1 — replaced the post-allocation freshness-desc sort with **round-robin interleaving across feed buckets**. **Why**: the freshness-desc sort I shipped in Phase 10 caused arXiv's hyper-fresh `published_at` timestamps to cluster at the top of the iteration, so when the user cancelled a run after 30 articles, all 30 were arXiv (14 cs.AI + 14 cs.CL + 2 cs.LG) and lab feeds got 0 — same starvation symptom as the original bug, just one step later in the pipeline. Round-robin now takes one article from each feed bucket per round, ordered by weight desc, so cancelling mid-run leaves every feed with at least `floor(processed / feedCount)` articles. Within each feed, articles stay newest-first via the per-feed query's `ORDER BY`. Memo updated (Q4 answer flipped). My original Q4 vote was wrong — caught on the first real test run via DB inspection of the cancelled run's distribution.
- **feat(analyze)**: Phase 10 — analyze allocation now distributes the batch across feeds by weight with spillover, replacing the broken `ORDER BY pub DESC LIMIT 100` query. **Why**: the first post-Phase-9 diagnostic run analyzed 99/100 articles from arXiv cs.AI alone while every lab feed (Anthropic, DeepMind, Meta AI, NVIDIA, Microsoft Research, Mistral, OpenAI) got zero coverage. Root cause: arXiv publishes hourly with hyper-fresh `published_at` timestamps that sort above every lab feed's days-old posts. **Fix**: new `feeds.analyze_weight` column (separate from `authority_score` per "one score, one thing"), pure `allocateSlots()` function with weighted proportional + spillover (12 unit tests), `analyzeUnanalyzed()` rewritten as a 4-phase execution (count per-feed → allocate → fetch-per-feed → sort across feeds by freshness → analyze). Per-feed allocation logged via `[analyze] allocation: feed=N/pending …` for run-time visibility. **Behavior change**: disabled feeds (`enabled = false`) now also skip analyze, not just fetch — matches the `analyze_weight = 0` semantics. Phase 10 design memo: [phase10-analyze-allocation-memo.md](phase10-analyze-allocation-memo.md). Validated by a realistic 12-feed allocation scenario test that mirrors today's actual production data — all 12 feeds get representation, no single arXiv feed exceeds 20 slots.
- **chore(pipeline)**: per-feed fetch logging restored after Task 44 dropped it. `[pipeline] run XXXXXXXX fetch:<feedName> added=N` info lines for successful feeds, `[pipeline] run XXXXXXXX fetch:<feedName> ERROR — <msg>` warn lines for failures. Restores visibility into specific feed errors (e.g. the still-unsolved Google AI Blog issue).
- **feat(feeds-ui)**: Settings → Feeds tab gains an "Analyze" weight column next to "Authority". Inline editable 0-1 input, same dirty-tracking + SaveBar pattern as the existing authority column. `AuthorityInput` generalized to a parameterized `WeightInput` component shared by both columns.
- **feat(shared)**: `feedSchema` extended with required `analyzeWeight: z.number().min(0).max(1)`. `createFeedSchema` and `updateFeedSchema` accept it as optional. Bonus fix: `createFeedSchema` now also accepts `authorityScore` on create (was previously PATCH-only — pre-existing gap).
- **fix(pipeline)**: orchestrator now re-checks the cancel flag after the summarize phase, before the final-status transition. **Why**: mid-summarize cancellation used to finalize the run as `completed` instead of `cancelled`. Root cause: summarize's in-service loop breaks correctly when the signal flag is flipped and returns partial counts, but the orchestrator had no post-Phase-3 check — so the flow fell through to `if (finalStatus === "running") finalStatus = "completed"`. Mid-analyze cancel was unaffected (Phase 3's entry check caught it), but mid-summarize had no entry check on anything after. Caught while writing Task 48 tests for per-article event threading; test 4 in that batch now locks the fix.

## 2026-04-13

- **fix(analyze)**: `analyzeUnanalyzed` now orders by `COALESCE(published_at, fetched_at) DESC` and filters to the last 14 days. **Why**: the old query had no `ORDER BY`, so Postgres returned rows in insertion order and the analyze batches burned entirely on OpenAI Blog (935 rows) then Hugging Face (762 rows), never reaching arXiv / Ars / MIT / VentureBeat. DB snapshot before the fix showed 6/9 feeds with 0 analyzed articles; after the filter, the pending queue fans out across those feeds (arXiv cs.AI 615, cs.LG 333, cs.CL 233, Ars 23, MIT 9, OpenAI 2 remaining in-window). Cutoff lives as `ANALYZE_MAX_AGE_DAYS = 14` at the top of `analyze.ts`.
- **chore(admin)**: `/pipeline/fetch` and `/pipeline/run-all` now emit one `[admin] fetch:<feedName> added=N` info line per feed, plus a `[admin] fetch:<feedName> ERROR — <msg>` warn when a feed throws. **Why**: the aggregate `feeds=N added=M errors=K` line couldn't tell us which specific feed was silent or failing.
- **ops**: Checked Google AI Blog (`https://blog.google/technology/ai/rss/`) manually — URL serves a 302 redirect to `https://blog.google/innovation-and-ai/technology/ai/rss/`, curl -L returns 200 + 31KB XML, and a direct `rss-parser.parseURL` call from node succeeds (20 items returned). The in-server fetches have nevertheless been silently failing since feed seed (`last_fetched_at = NULL`, all 8 other feeds successfully fetched at 01:30 today). **Status**: pending. The per-feed fetch logging added above will surface the actual in-server error on the next `/pipeline/fetch` tick. Do not guess-fix before we have the log line.
- **decision**: Phase 8 tasks 37 (retention policy) and 38 (ranked recency filter) closed as won't-do. **Why**: the analyze-side 14-day filter already owns retention behaviorally, and `/ranked` already owns recency via the composite freshness weight. Adding a second mechanism for either would create edge effects and duplicate state. If dashboard still leans on old articles after the analyze fan-out, tune `freshness_lambda` in settings — runtime-tunable, no code change. Revisit retention only if row counts become a real disk concern (currently ~2500 unanalyzed historical rows, negligible).

- **plan(phase-9)**: Pipeline observability — decisions locked, implementation not yet started. Chosen transport: Server-Sent Events (single `GET /admin/pipeline/stream`), not client-side phase orchestration (too coarse) and not WebSockets (one-way channel). Chosen cancel semantics: in-memory flag checked between LLM calls, in-flight call completes normally, partial results retained. Chosen architecture: single `runPipelineWithProgress` orchestrator shared by both manual trigger and scheduler cron — both paths write to a new `pipeline_runs` table for unified history. Explicitly out of scope: job queue, WebSockets, retry logic, pause/resume, multi-user attribution. Full rationale in [phase9-observability-memo.md](phase9-observability-memo.md); visual reference in [phase9-pipeline-ops-mockup.html](phase9-pipeline-ops-mockup.html) (open in browser to see the three production states — idle, running, history-open — rendered in the newsroom-workstation aesthetic). Implementation order is tasks 39 → 49 and the dependency chain is strict: schema → orchestrator → service threading → scheduler refactor → cron util → API → frontend → tests. No parallel work.
