## 2026-04-13

- **fix(analyze)**: `analyzeUnanalyzed` now orders by `COALESCE(published_at, fetched_at) DESC` and filters to the last 14 days. **Why**: the old query had no `ORDER BY`, so Postgres returned rows in insertion order and the analyze batches burned entirely on OpenAI Blog (935 rows) then Hugging Face (762 rows), never reaching arXiv / Ars / MIT / VentureBeat. DB snapshot before the fix showed 6/9 feeds with 0 analyzed articles; after the filter, the pending queue fans out across those feeds (arXiv cs.AI 615, cs.LG 333, cs.CL 233, Ars 23, MIT 9, OpenAI 2 remaining in-window). Cutoff lives as `ANALYZE_MAX_AGE_DAYS = 14` at the top of `analyze.ts`.
- **chore(admin)**: `/pipeline/fetch` and `/pipeline/run-all` now emit one `[admin] fetch:<feedName> added=N` info line per feed, plus a `[admin] fetch:<feedName> ERROR — <msg>` warn when a feed throws. **Why**: the aggregate `feeds=N added=M errors=K` line couldn't tell us which specific feed was silent or failing.
- **ops**: Checked Google AI Blog (`https://blog.google/technology/ai/rss/`) manually — URL serves a 302 redirect to `https://blog.google/innovation-and-ai/technology/ai/rss/`, curl -L returns 200 + 31KB XML, and a direct `rss-parser.parseURL` call from node succeeds (20 items returned). The in-server fetches have nevertheless been silently failing since feed seed (`last_fetched_at = NULL`, all 8 other feeds successfully fetched at 01:30 today). **Status**: pending. The per-feed fetch logging added above will surface the actual in-server error on the next `/pipeline/fetch` tick. Do not guess-fix before we have the log line.
- **decision**: Phase 8 tasks 37 (retention policy) and 38 (ranked recency filter) closed as won't-do. **Why**: the analyze-side 14-day filter already owns retention behaviorally, and `/ranked` already owns recency via the composite freshness weight. Adding a second mechanism for either would create edge effects and duplicate state. If dashboard still leans on old articles after the analyze fan-out, tune `freshness_lambda` in settings — runtime-tunable, no code change. Revisit retention only if row counts become a real disk concern (currently ~2500 unanalyzed historical rows, negligible).
- **fix(feeds)**: Google AI Blog URL was `https://blog.google/technology/ai/rss/` which 302-redirects to `https://blog.google/innovation-and-ai/technology/ai/rss/`. In the long-lived server rss-parser threw `Cannot read properties of null (reading 'constructor')` on the redirect path (surfaced by the new per-feed logging). Patched to the canonical target via `seed-new-sources-2026-04-13.ts`.
- **feat(feeds)**: Added 6 new sources after verifying each URL produced parseable RSS with fresh items:
  - **Lab (authority 0.6)**: DeepMind (`deepmind.google/blog/rss.xml`), NVIDIA Developer (`developer.nvidia.com/blog/feed/`), Microsoft Research (`microsoft.com/en-us/research/blog/feed/`).
  - **News aggregator (authority 0.3)**: Google News RSS queries for Anthropic, Meta AI, and Mistral AI — used because none of the three expose a first-party RSS endpoint. Lower authority reflects 3rd-party/press noise. xAI GNews was tested but dropped because sample items were ~70% litigation/controversy drama and ~30% actual news.
  - Rejected candidates documented: Anthropic direct RSS (no endpoint exists), Meta AI direct RSS (404 on all known paths), Mistral direct RSS (same), Cohere RSS (malformed XML), Meta Research blog (last post 2023), NVIDIA AI deep-learning category feed (last post 2026-01-05 — dev blog is strictly better).
  - Seed script `apps/api/src/db/seed-new-sources-2026-04-13.ts` is idempotent (onConflictDoNothing on URL) and safe to re-run.
- **chore(feeds)**: VentureBeat AI disabled in the same migration — newest article was 2026-01-22, feed has been static since seed. 7 total articles, 0 reached analyze. Left `enabled=false` rather than deleted so the row + articles stay for history.
- **refactor(feeds)**: Upgraded Anthropic / Meta AI / Mistral AI Google News feeds from keyword queries to `site:<domain>` queries. **Why**: the original `"anthropic" claude` etc. queries returned mostly 3rd-party press coverage. `site:anthropic.com` returns Anthropic's *own* engineering, research, and news posts — "Trustworthy agents in practice", "Claude Mythos System Card", "Scaling Managed Agents", etc. These are now effectively first-party crawls, so authority bumped 0.3 → 0.5 and category moved from `news-aggregator` to `lab-proxy`. Same upgrade applied to Meta AI (Muse Spark, TRIBE v2 research releases) and Mistral AI. Follow-up migration `seed-upgrade-sites-2026-04-13.ts`. Prompted by the question "does Anthropic have technical articles? are those in our source?" — yes, and the keyword-query version was missing them.
- **plan(phase-9)**: Pipeline observability — decisions locked, implementation not yet started. Chosen transport: Server-Sent Events (single `GET /admin/pipeline/stream`), not client-side phase orchestration (too coarse) and not WebSockets (one-way channel). Chosen cancel semantics: in-memory flag checked between LLM calls, in-flight call completes normally, partial results retained. Chosen architecture: single `runPipelineWithProgress` orchestrator shared by both manual trigger and scheduler cron — both paths write to a new `pipeline_runs` table for unified history. Explicitly out of scope: job queue, WebSockets, retry logic, pause/resume, multi-user attribution. Full rationale in [phase9-observability-memo.md](phase9-observability-memo.md); visual reference in [phase9-pipeline-ops-mockup.html](phase9-pipeline-ops-mockup.html) (open in browser to see the three production states — idle, running, history-open — rendered in the newsroom-workstation aesthetic). Implementation order is tasks 39 → 49 and the dependency chain is strict: schema → orchestrator → service threading → scheduler refactor → cron util → API → frontend → tests. No parallel work.
